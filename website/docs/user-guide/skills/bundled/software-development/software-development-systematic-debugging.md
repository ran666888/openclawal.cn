# 系统调试

## 概述

随机修复会浪费时间并产生新的错误。快速补丁掩盖了根本问题。

**核心原则：** 在尝试修复之前始终找到根本原因。症状修复失败。

**违反此过程的字面意思就是违反调试精神。**

## 铁律

````
不首先调查根本原因就无法修复
````

如果您尚未完成第一阶段，则无法提出修复建议。

## 何时使用

用于任何技术问题：
- 测试失败
- 生产中的错误
- 意外行为
- 性能问题
- 构建失败
- 整合问题

**特别是在以下情况下使用此功能：**
- 在时间压力下（紧急情况让人很容易猜测）
- “只需一个快速解决方案”似乎是显而易见的
- 您已经尝试过多次修复
- 之前的修复不起作用
- 你没有完全理解这个问题

**在以下情况下不要跳过：**
- 问题看起来很简单（简单的错误也有根本原因）
- 你很着急（着急保证返工）
- 有人希望现在就修复它（系统化比颠簸更快）

## 四个阶段

您必须先完成每个阶段，然后才能进入下一阶段。

---

## 第一阶段：根本原因调查

**尝试任何修复之前：**

### 1. 仔细阅读错误消息

- 不要跳过过去的错误或警告
- 它们通常包含精确的解决方案
- 完整读取堆栈跟踪
- 记录行号、文件路径、错误代码

**操作：** 在相关源文件上使用“read_file”。使用“search_files”在代码库中查找错误字符串。

### 2. 一致地再现

- 你能可靠地触发它吗？
- 具体步骤是什么？
- 每次都会发生吗？
- 如果不可重现→收集更多数据，不要猜测

**操作：** 使用 `terminal` 工具运行失败的测试或触发错误：

````bash
# 运行特定的失败测试
pytest 测试/test_module.py::test_name -v

# 以详细输出运行
pytest 测试/test_module.py -v --tb=long
````

### 3.检查最近的更改

- 是什么变化导致了这种情况？
- Git diff，最近的提交
- 新的依赖项，配置更改

**行动：**

````bash
# 最近提交
git log --oneline -10

# 未提交的更改
git 差异

# 特定文件的更改
git log -p --follow src/problematic_file.py | git log -p --follow src/problematic_file.py |头-100
````

### 4. 收集多组件系统中的证据

**当系统有多个组件时（API → 服务 → 数据库、CI → 构建 → 部署）：**

**在提出修复建议之前，添加诊断工具：**

对于每个组件边界：
- 记录进入组件的数据
- 记录退出组件的数据
- 验证环境/配置传播
- 检查每一层的状态

运行一次以收集证据以显示其损坏的位置。
然后分析证据来识别故障组件。
然后调查该特定组件。

### 5. 跟踪数据流

**当错误深入调用堆栈时：**

- 不良价值从何而来？
- 什么叫这个具有错误值的函数？
- 不断向上游追踪，直到找到源头
- 从根源上解决，而不是从症状上解决

**操作：** 使用 `search_files` 来跟踪引用：

````蟒蛇
# 找到函数被调用的地方
search_files("function_name(", path="src/", file_glob="*.py")

# 找到变量设置的位置
search_files("variable_name\\s*=", path="src/", file_glob="*.py")
````

### 第一阶段完成清单

- [ ] 错误消息已完全阅读并理解
- [ ] 问题一致重现
- [ ] 最近确定和审查的变更
- [ ] 收集的证据（日志、状态、数据流）
- [ ] 问题与特定组件/代码隔离
- [ ] 形成根本原因假设

**停止：** 在您了解发生这种情况的原因之前，请勿继续进行第 2 阶段。

---

## 第二阶段：模式分析

**修复前先找到图案：**

### 1. 查找工作示例

- 在同一代码库中找到相似的工作代码
- 有哪些作品与已损坏的作品相似？

**操作：** 使用 `search_files` 查找可比较的模式：

````蟒蛇
search_files("similar_pattern", path="src/", file_glob="*.py")
````

### 2. 与参考文献进行比较

- 如果实现模式，请完整阅读参考实现
- 不要略读——阅读每一行
- 申请前充分了解图案

### 3. 找出差异

- 正常工作和坏掉有什么区别？
- 列出每一个差异，无论多么小
- 不要认为“那不重要”

### 4.了解依赖关系

- 这还需要什么其他组件？
- 什么设置、配置、环境？
- 它做出了什么假设？

---

## 第三阶段：假设和测试

**科学方法：**

### 1. 形成单一假设

- 明确说明：“我认为 X 是根本原因，因为 Y”
- 写下来
- 要具体，不要含糊

### 2. 最少测试

- 做出尽可能小的改变来检验假设
- 一次一个变量
- 不要一次修复多个问题

### 3. 继续之前进行验证

- 有效果吗？ → 第四阶段
- 没工作？ → 形成新的假设
- 不要在上面添加更多修复

### 4. 当你不知道时

- 说“我不明白X”
- 别假装知道
- 向用户寻求帮助
- 研究更多

---

## 第四阶段：实施

**解决根本原因，而不是症状：**

### 1. 创建失败的测试用例

- 最简单的再现
- 如果可能的话进行自动化测试
- 修复前必须有
- 使用“测试驱动开发”技能

### 2. 实施单一修复

- 解决已确定的根本原因
- 一次更改一项
- 没有“当我在这里”的改进
- 没有捆绑重构

### 3. 验证修复

````bash
# 运行具体的回归测试
pytest 测试/test_module.py::test_regression -v

# 运行全套——没有回归
pytest 测试/-q
````

### 4. 如果修复不起作用 — 三法则

- **停止。**
- 计数：您尝试过多少次修复？
- 如果 < 3：返回阶段 1，用新信息重新分析
- **如果 ≥ 3：停止并质疑架构（下面的步骤 5）**
- 在没有进行架构讨论的情况下，不要尝试修复 #4

### 5. 如果 3 个以上修复失败：问题架构

**指示架构问题的模式：**
- 每个修复都会在不同的地方揭示新的共享状态/耦合
- 修复需要“大规模重构”才能实施
- 每次修复都会在其他地方产生新的症状

**停下来询问基本原理：**
- 这种模式从根本上来说合理吗？
- 我们是“纯粹因为惯性而坚持下去”吗？
- 我们应该重构架构还是继续修复症状？

**在尝试更多修复之前与用户讨论。**

这不是一个失败的假设——这是一个错误的架构。

---

## 危险信号 — 停止并遵循流程

如果你发现自己在想：
- “现在快速修复，稍后调查”
- “尝试改变 X 看看是否有效”
- “添加多个更改，运行测试”
- “跳过测试，我手动验证”
- “可能是 X，让我解决这个问题”
- “我不完全明白，但这可能有用”
- “模式说 X 但我会以不同的方式调整它”
- “以下是主要问题：[列出未经调查的修复]”
- 在追踪数据流之前提出解决方案
- **“再尝试一次修复”（当已尝试 2+ 次时）**
- **每次修复都会在不同的地方揭示一个新问题**

**所有这些都意味着：停止。返回第一阶段。**

**如果 3 个以上修复失败：** 质疑架构（第 4 阶段第 5 步）。

## 常见的合理化

|对不起|现实|
|--------|---------|
| “问题很简单，不需要流程”|简单的问题也有根本原因。对于简单的错误，处理速度很快。 |
| “紧急情况，没有时间处理”|系统调试比猜测和检查颠簸更快。 |
| “先尝试一下，然后再调查” |第一个修复设置了模式。从一开始就做对。 |
| “确认修复有效后我将编写测试” |未经测试的修复不会持续下去。首先测试证明这一点。 |
| “一次进行多个修复可以节省时间”|无法隔离有效的方法。导致新的错误。 |
| “参考资料太长，我会调整模式”|部分理解肯定会出现错误。完整地阅读它。 |
| “我看到问题了，让我解决它”|看到症状≠了解根本原因。 |
| “再尝试一次修复”（两次以上失败后）| 3+次失败=架构问题。对模式提出质疑，不要再次修复。 |

## OpenClaw 代理集成

### 调查工具

在第一阶段使用这些 OpenClaw 工具：

- **`search_files`** — 查找错误字符串、跟踪函数调用、定位模式
- **`read_file`** — 读取带有行号的源代码以进行精确分析
- **`terminal`** — 运行测试，检查 git 历史记录，重现错误
- **`web_search`/`web_extract`** — 研究错误消息、库文档

### 使用 delegate_task

对于复杂的多组件调试，调度调查子代理：

````蟒蛇
委托任务（
    goal="调查[特定测试/行为]失败的原因",
    上下文=“”“
    遵循系统调试技巧：
    1.仔细阅读错误信息
    2. 重现问题
    3. 追踪数据流向，找出根本原因
    4. 报告调查结果——暂不修复

    错误：[粘贴完整错误]
    文件：[失败代码的路径]
    测试命令：[确切命令]
    """,
    工具集=['终端', '文件']
）
````

### 通过测试驱动开发

修复错误时：
1. 编写一个重现错误的测试（红色）
2. 系统调试，查找根本原因
3. 修复根本原因（绿色）
4. 测试证明修复并防止回归

## 现实世界的影响

从调试会话：
- 系统方法：15-30 分钟修复
- 随机修复方法：2-3小时的颠簸
- 首次修复率：95% vs 40%
- 引入新错误：接近零与常见

**没有捷径。无需猜测。系统化总是胜利。**
