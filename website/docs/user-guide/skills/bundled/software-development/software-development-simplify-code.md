# 简化代码——并行审查和清理

由三位专注的审阅者来审查您最近的代码更改
并行，汇总他们的发现，并应用值得应用的修复程序。

**核心原则：** 三名狭义审稿人击败一名广泛审稿人。每一个
深入搜索代码库以找出一类问题——重用、质量、
效率——而不分散对这三个方面的注意力。他们跑
同时，因此您需要支付一篇评论的延迟，而不是三篇评论。

## 何时使用

当用户说出以下任一内容时触发此技能：

- “简化”/“简化我的更改”/“简化这些更改”
- “查看我的代码”/“查看我最近的更改”/“清理我的更改”
- “/simplify”（如果他们继承了克劳德代码习惯）

用户可以添加的可选修饰符 - 尊重它们：

- **焦点：**“简化对效率的关注”→仅运行效率审核器
  （或对聚合进行加权）。公认的焦点：“重用”，
  “质量”、“效率”。
- **试运行：**“简化但不改变任何东西”/“只是报告”→运行
  三位审稿人，目前的调查结果，不适用。申请前先询问。
- **范围：**“简化最后一次提交”/“简化分阶段”/“简化
  src/foo.py" → 相应地缩小 diff 源范围（参见阶段 1）。

每次编辑后不要自动运行它。它花费了三个子代理的价值
令牌——仅当用户明确询问时才调用它。

## 过程

### 第 1 阶段 — 识别变化

捕获差异以进行审查。根据用户要求选择来源，在
这个默认顺序：

````bash
# 1.默认值：未提交的工作树更改（跟踪文件）
git 差异

# 2. 如果为空，请包含分阶段更改
git diff 头

# 3. 用户可能请求的范围变体：
git diff --staged #“分阶段更改”
git diff HEAD~1 #“最后一次提交”
git diff main...HEAD #“这个分支”/“我的 PR”
git diff -- src/foo.py # 特定文件
````

如果`git diff`和`git diff HEAD`都是空的并且没有git repo或者没有
更改，回退到用户明确命名的文件或
最近在此会话中创建/编辑。如果你实在找不到
更改代码，说出来并停止 - 没有什么可以简化的。

捕获完整的差异文本。注意它的大小：如果它非常大（比如 >2000
更改行），警告用户三个子代理各自携带完整的差异
将是大量令牌，并提供缩小范围（每个目录，每个提交）
在继续之前。

### 第 2 阶段 — 并行启动三名审稿人

使用 `delegate_task` **批处理模式** — 将所有三个任务传递到一个 `tasks` 中
数组，以便它们同时运行。三是该模式的正确扇出；
它完全在“delegation.max_concurrent_children”预算范围内
默认安装。

为**每个**审阅者提供**完整的差异**（不是片段 - 跨文件
问题隐藏在间隙中）加上绝对的存储库路径，以便他们可以搜索
更广泛的代码库。每个审阅者都会获得“终端”、“文件”和“搜索”
工具集（因此可以使用“git”、“read_file”和“search_files”/grep）。

告诉每位审稿人：
- 在现有代码库中搜索证据（不要仅根据差异进行推理）。
- **应用切斯特顿的栅栏：** 在标记任何要删除的内容之前，运行
  上线`gitblame`就可以理解它为什么存在。如果你无法确定
  最初的目的，标记为“信心：低”——不要猜测。
- 以结构化输出的形式报告调查结果，充满信心和风险：
  ````
  文件：行 → 问题 → 建议修复 |置信度：高/中/低 |风险：安全/小心/有风险
  ````
  - **SAFE** = 证明不会影响行为（未使用的导入，注释掉
    代码、传递包装器）。自动应用这些。
  - **小心** = 在不改变语义的情况下进行改进（重命名局部变量，
    展平嵌套三元，提取助手）。通过测试验证进行申请。
  - **有风险** = 可能会改变行为或违反公共合同（N+1
    重组、公共 API 重命名、内存生命周期更改）。标记为
    人工审核——不要自动应用。
- 跳过尼特和风格的流失。只标记有实质性改进的事情
  代码。

实现这三个目标（删除用户关注的任何目标除外）：

**审阅者 1 — 代码重用**
> 查看此 diff 中是否存在重复功能的代码
> 代码库。搜索实用程序模块、共享帮助程序和相邻文件
> （使用 search_files / grep）现有函数、常量或模式
> 新代码可以调用而不是重新实现。标志：新功能
> 重复现有的；现有实用程序的手工逻辑
> 已经做了（手动字符串/路径操作、自定义环境检查、临时
> 类型保护，重新实现解析）。对于每个，将现有的事物命名为
> 使用及其所在位置。

**审阅者 2 — 代码质量**
> 检查此差异是否存在质量问题。寻找：冗余状态（值
> 重复或可以从现有状态派生；没有的缓存
> 需要存在）；参数蔓延（新参数固定在函数的位置
> 应该进行重组）；带有变化的复制粘贴（几乎重复
> 应该共享抽象的块）；泄漏抽象（暴露
> 内部结构，打破现有的封装边界）；字符串类型
> 代码（常量/枚举/注册表已存在的原始字符串 - 检查
> 标记前的规范注册表）； AI 生成的倾斜模式（额外
> 注释重述了明显的代码，例如 `count++` 之上的 `//increment counter`；
> 对已经验证的输入进行不必要的防御性空检查； `如任何`
> 绕过类型系统的强制转换；与其余部分不一致的模式
> 文件）。对于每个，给出具体的重构。

**审稿人 3 — 效率**
> 检查此差异是否存在效率问题。寻找：不必要的工作
>（冗余计算、重复文件读取、重复API调用、N+1
> 访问模式）；错过并发（独立操作按顺序运行）；
> 热路径膨胀（启动或每个请求路径上的繁重/阻塞工作）；
> TOCTOU 反模式（在操作之前预先检查存在性，而不是执行
> 操作和处理错误）；内存问题（无限增长、缺失
> 清理、侦听器/处理泄漏）；读取范围过于广泛（加载整个文件
> 什么时候切片就可以了）；无提示失败（空 catch 块、忽略错误
> 返回，` except: pass`，`.catch(() => {})`，没有处理，错误
> 传播间隙——这些隐藏错误并且应该至少在之前记录
> 吞咽）。对于每个问题，给出具体的解决方案以及为什么它更快或更安全。

### 第 3 阶段 — 聚合和应用

等待所有三个返回（批处理模式将它们一起返回）。

1. **将**结果合并到一个列表中，删除审阅者重叠的地方的重复内容。
2. **丢弃误报** — 您拥有最多的上下文；你不必
   与审稿人争论，只是默默地放弃薄弱或错误的建议。
3. **解决冲突。** 审阅者可以不同意（审阅者 1：“使用现有的
   util X”；评论者 3：“X 很慢，内联它”）。默认解析顺序：
   **正确性 > 用户声明的焦点 > 可读性/重用性 > 微性能。**
   除非路径确实正确，否则不要应用有损清晰度的性能“修复”
   热。当两个建议相互排斥并且都有道理时，选择
   涉及较少代码的一个并注意替代方案。
4. **按风险等级顺序申请：**
   - **安全第一**（自动应用）：未使用的导入，注释掉的代码，
     传递包装器、冗余类型断言。之后运行测试。
   - **下一步小心**（应用验证，一次一个文件）：重命名
     当地人，压平三元，提取助手，巩固欺骗。运行测试
     在每个文件之后。恢复任何损坏的部分。
   - **最后有风险**（审查标记 - 请勿自动申请）：N+1 重组，
     公共 API 更改、并发修复、错误处理更改。现在
     每个都有风险描述和测试覆盖率状态。
   如果用户选择试运行，则显示所有三个层并且不应用任何内容。
5. **验证**您没有破坏任何东西：运行项目的目标测试
   触摸的文件（不是完整的套件），并重新运行任何 linter/类型检查
   回购协议使用。如果某个修复破坏了测试，请恢复该修复并报告它。
6. **总结**您更改的内容：已应用修复的简短列表，按以下分组
   审阅者类别和风险等级，以及您故意跳过的任何发现
   以及为什么。

## 陷阱

- **不要扇出超过 ~3。** 更多的审稿人意味着更多的成本和更多
  相互矛盾的建议需要调和，而不是更好的覆盖。三大类
  覆盖空间。
- **将整个差异提供给每个审阅者。** 在审阅者之间拆分差异
  破坏了设计——跨文件复制和 N+1 只出现在
  完整图片。
- **审稿人搜索，他们不会猜测。** 没有指向的重用发现
  现有的实用程序（“可能有一个帮手”）是噪音。要求
  `文件：行`证据；放弃缺乏它的发现。
- **应用≠重写。** 这是对用户最近更改的清理，而不是
  重构整个模块的许可。将编辑范围限制在差异范围内
  触摸加上修复所需的最小周围环境变化。
- **尊重项目约定。** 如果仓库有 AGENTS.md / CLAUDE.md /
  HERMES.md 或 linter 配置，将这些规则折叠到审阅者提示中，以便
  建议与房屋风格相匹配，而不是与之对抗。
- **大的差异会影响上下文。**如果差异很大，请先缩小范围
  委托 — 三个子代理，每个子代理携带 5000 行 diff 是昂贵的，并且
  可能会被截断。
- **过度信任死代码工具。** `knip`、`ts-prune` 和 `depcheck` 标志
  动态使用的导出（基于字符串的导入、反射）。总是
  在删除之前使用 grep 查找符号名称 - 干净的工具报告并不能证明。
- **在不检查公共合约的情况下重命名。** 导出名称、API 路由
  路径、数据库列名称和配置键都是契约——即使名称是
  不好，重命名破坏了消费者。将公共合同变更标记为“有风险”；从不
  自动重命名它们。
- **删除“不必要的”错误处理。**空的 catch 块或被忽略
  错误可能是故意的——该错误是预期的并且是良性的
  上下文。标记它，不要删除它；让人类来决定。
