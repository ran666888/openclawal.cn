# 计划模式

当用户想要计划而不是执行时，请使用此技能。

## 核心行为

对于这一轮，你只是在计划。

- 不执行代码。
- 不要编辑除计划 Markdown 文件之外的项目文件。
- 不要运行变异终端命令、提交、推送或执行外部操作。
- 需要时，您可以使用只读命令/工具检查存储库或其他上下文。
- 您的交付成果是保存在“.hermes/plans/”下活动工作区中的降价计划。

## 输出要求

写一个具体且可操作的降价计划。

相关时包括：
- 目标
- 当前背景/假设
- 提议的方法
- 分步计划
- 文件可能会改变
- 测试/验证
- 风险、权衡和悬而未决的问题

如果任务与代码相关，请包括确切的文件路径、可能的测试目标和验证步骤。

## 保存位置

使用“write_file”保存计划：
- `.hermes/plans/YYYY-MM-DD_HHMMSS-<slug>.md`

将其视为相对于活动工作目录/后端工作区。 OpenClaw 文件工具是后端感知的，因此使用此相对路径可以使计划与本地、docker、ssh、modal 和 daytona 后端上的工作区保持一致。

如果运行时提供特定目标路径，请使用该确切路径。
如果没有，请自己在“.hermes/plans/”下创建一个合理的带时间戳的文件名。

## 交互风格

- 如果要求足够明确，直接写出计划。
- 如果没有明确的指令伴随“/plan”，则从当前对话上下文推断任务。
- 如果确实未明确说明，请提出一个简短的澄清问题，而不是猜测。
- 保存计划后，简短回复您的计划内容和保存的路径。

# 写好计划

这项技能的其余部分是编写“好的”实施计划的技巧——上面的 markdown 文件中的内容。

## 概述

假设实施者对代码库的背景为零并且品味有问题，则编写全面的实施计划。记录他们需要的一切：要接触哪些文件、完整的代码、测试命令、要检查的文档、如何验证。给他们分配一些小任务。干燥。亚格尼。时分驱动。频繁提交。

假设实施者是一位熟练的开发人员，但对工具集或问题领域几乎一无所知。假设他们不太了解良好的测试设计。

**核心原则：** 好的计划，执行起来显而易见。如果必须有人猜测，那么这个计划就是不完整的。

## 当完整的实施计划有帮助时

**始终在以下时间之前使用：**
- 实现多步骤功能
- 分解复杂的需求
- 通过子代理驱动开发委托给子代理

**在以下情况下不要跳过：**
- 功能看起来很简单（假设会导致错误）
- 你打算自己实现（未来你需要指导）
- 单独工作（文件很重要）

## 小任务粒度

**每项任务 = 2-5 分钟的专注工作。**

每一步都是一个动作：
- “编写失败的测试” — 步骤
- “运行它以确保它失败” - 步骤
- “实现最少的代码以使测试通过” - 步骤
- “运行测试并确保它们通过” - 步骤
- “提交”——步骤

**太大：**
``降价
### 任务1：构建认证系统
[跨 5 个文件 50 行代码]
````

**合适的尺寸：**
``降价
### 任务 1：创建包含电子邮件字段的用户模型
[10 行，1 个文件]

### 任务 2：向用户添加密码哈希字段
[8行，1个文件]

### 任务 3：创建密码哈希实用程序
[15行，1个文件]
````

## 规划文档结构

### 标题（必填）

每个计划必须以以下内容开始：

``降价
# [功能名称]实施计划

> **对于 OpenClaw：** 使用子代理驱动的开发技能来逐个任务地实施该计划。

**目标：** [用一句话描述其构建内容]

**架构：** [关于方法的 2-3 句话]

**技术堆栈：** [关键技术/库]

---
````

### 任务结构

每个任务都遵循以下格式：

````降价
### 任务 N：[描述性名称]

**目标：** 这项任务要完成什么（一句话）

**文件：**
- 创建：`exact/path/to/new_file.py`
- 修改：`exact/path/to/existing.py:45-67`（行号，如果已知）
- 测试：`tests/path/to/test_file.py`

**第 1 步：编写失败的测试**

````蟒蛇
def test_specific_behavior():
    结果 = 函数（输入）
    断言结果==预期
````

**步骤 2：运行测试以验证失败**

运行：`pytest测试/路径/test.py::test_specific_behavior -v`
预期：FAIL —“函数未定义”

**第 3 步：编写最小实现**

````蟒蛇
def 函数（输入）：
    预期回报
````

**第 4 步：运行测试以验证通过**

运行：`pytest测试/路径/test.py::test_specific_behavior -v`
预期：通过

**第 5 步：承诺**

````bash
git add 测试/路径/test.py src/path/file.py
git commit -m“壮举：添加特定功能”
````

## 写作过程

### 第 1 步：了解需求

阅读并理解：
- 功能要求
- 设计文档或用户描述
- 验收标准
- 限制

### 第 2 步：探索代码库

使用Hermes工具了解项目：

````蟒蛇
# 了解项目结构
search_files（“*.py”，目标=“文件”，路径=“src /”）

# 查看相似的特征
search_files("similar_pattern", path="src/", file_glob="*.py")

# 检查现有测试
search_files(“*.py”，目标=“文件”，路径=“测试/”)

# 读取关键文件
read_file("src/app.py")
````

### 步骤 3：设计方法

决定：
- 架构模式
- 文件组织
- 需要依赖
- 测试策略

### 第 4 步：编写任务

按顺序创建任务：
1. 设置/基础设施
2. 核心功能（每个功能的TDD）
3. 边缘情况
4. 整合
5. 清理/记录

### 第 5 步：添加完整详细信息

对于每项任务，包括：
- **确切的文件路径**（不是“配置文件”，而是“src/config/settings.py”）
- **完整的代码示例**（不是“添加验证”，而是实际代码）
- **精确命令**与预期输出
- **验证步骤**证明任务有效

### 第 6 步：审查计划

检查：
- [ ] 任务是连续且符合逻辑的
- [ ] 每个任务都很短（2-5 分钟）
- [ ] 文件路径准确
- [ ] 代码示例完整（可复制粘贴）
- [ ] 命令与预期输出完全一致
- [ ] 没有缺失上下文
- [ ] 应用 DRY、YAGNI、TDD 原则

## 原则

### DRY（不要重复自己）

**不好：** 在 3 个地方进行复制粘贴验证
**好：** 提取验证功能，随处使用

### YAGNI（你不需要它）

**坏：**为未来的需求添加“灵活性”
**好：** 仅实施现在需要的内容

````蟒蛇
# 不好 — YAGNI 违规
班级用户：
    def __init__(自我，姓名，电子邮件)：
        self.name = 名字
        self.email = 电子邮件
        self.preferences = {} # 还不需要！
        self.metadata = {} # 还不需要！

# 好 — 亚格尼
班级用户：
    def __init__(自我，姓名，电子邮件)：
        self.name = 名字
        self.email = 电子邮件
````

### TDD（测试驱动开发）

每个生成代码的任务都应该包含完整的 TDD 周期：
1. 编写失败的测试
2.运行验证失败
3. 编写最少的代码
4.运行验证通过

有关详细信息，请参阅“测试驱动开发”技能。

### 频繁提交

每项任务后提交：
````bash
git add [文件]
git commit -m“类型：描述”
````

## 常见错误

### 模糊任务

**坏：**“添加身份验证”
**好：**“使用电子邮件和密码哈希字段创建用户模型”

### 不完整的代码

**不好：**“第1步：添加验证功能”
**好：**“第1步：添加验证功能”后面是完整的功能代码

### 缺少验证

**不好：**“第 3 步：测试它是否有效”
**好：**“第3步：运行`pytest测试/test_auth.py -v`，预期：3通过”

### 缺少文件路径

**坏：**“创建模型文件”
**好：**“创建：`src/models/user.py`”

## 执行交接

保存计划后，给出执行方式：

**“计划已完成并保存。准备使用子代理驱动开发来执行 - 我将为每个任务分派一个新的子代理并进行两阶段审查（规范合规性然后是代码质量）。我应该继续吗？”**

执行时，使用`subagent-driven-development`技能：
- 每个任务都有新鲜的“delegate_task”，具有完整的上下文
- 每项任务后的规范合规性审查
- 规范通过后的代码质量审查
- 仅当两项审查均批准后才继续

## 记住

````
一口大小的任务（每个 2-5 分钟）
确切的文件路径
完整代码（可复制粘贴）
具有预期输出的精确命令
验证步骤
干、YAGNI、TDD
频繁提交
````

**好的计划使实施变得显而易见。**
