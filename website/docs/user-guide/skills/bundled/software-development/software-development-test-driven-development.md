# 测试驱动开发 (TDD)

## 概述

先写测试。看着它失败。编写最少的代码即可通过。

**核心原则：** 如果你没有看到测试失败，你就不知道它是否测试了正确的东西。

**违反规则的字面意思就是违反规则的精神。**

## 何时使用

**总是：**
- 新功能
- 错误修复
- 重构
- 行为改变

**例外情况（先询问用户）：**
- 一次性原型
- 生成的代码
- 配置文件

想“跳过 TDD 这一次”吗？停止。这就是合理化。

## 铁律

````
如果没有首先进行失败的测试，就没有生产代码
````

测试前先写代码？删除它。重新开始。

**没有例外：**
- 不要将其保留为“参考”
- 编写测试时不要“适应”它
- 别看它
- 删除就是删除

实施新的测试。时期。

## 红-绿-重构循环

### 红色 — 编写失败测试

编写一个最小的测试来显示应该发生什么。

**良好的测试：**
````蟒蛇
def test_retries_failed_operations_3_times():
    尝试 = 0
    定义操作（）：
        非本地尝试
        尝试 += 1
        如果尝试 < 3：
            引发异常（'失败'）
        返回“成功”

    结果 = 重试操作（操作）

    断言结果=='成功'
    断言尝试 == 3
````
清晰的名字，测试真实的行为，一件事。

**糟糕的测试：**
````蟒蛇
def test_retry_works():
    模拟 = MagicMock()
    mock.side_effect = [Exception(), Exception(), '成功']
    结果=重试操作（模拟）
    assert result == 'success' # 重试次数怎么样？定时？
````
模糊的名称，测试模拟而不是真实的代码。

**要求：**
- 每次测试一种行为
- 清晰的描述性名称（名称中的“和”？将其分开）
- 真实代码，而不是模拟（除非确实不可避免）
- 名称描述的是行为，而不是实现

### 验证红色 — 观察失败

**强制。切勿跳过。**

````bash
# 使用终端工具运行具体测试
pytest 测试/test_feature.py::test_specific_behavior -v
````

确认：
- 测试失败（不是拼写错误导致的错误）
- 预计会出现失败消息
- 由于缺少该功能而失败

**测试立即通过？** 您正在测试现有行为。修复测试。

**测试错误？** 修复错误，重新运行，直到正确失败。

### 绿色 — 最少代码

编写最简单的代码来通过测试。而已。

**好：**
````蟒蛇
def 添加（a，b）：
    return a + b # 没有额外的内容
````

**不好：**
````蟒蛇
def 添加（a，b）：
    结果 = a + b
    logging.info(f"添加 {a} + {b} = {result}") # 额外！
    返回结果
````

不要添加功能、重构其他代码或在测试之外进行“改进”。

**作弊在绿色中是可以的：**
- 硬编码返回值
- 复制粘贴
- 重复代码
- 跳过边缘情况

我们将在 REFACTOR 中修复它。

### 验证绿色 — 观察它通过

**强制。**

````bash
# 运行具体测试
pytest 测试/test_feature.py::test_specific_behavior -v

# 然后运行所有测试来检查回归
pytest 测试/-q
````

确认：
- 测试通过
- 其他测试仍然通过
- 输出原始（没有错误、警告）

**测试失败？** 修复代码，而不是测试。

**其他测试失败？** 立即修复回归。

### 重构 — 清理

仅绿色之后：
- 删除重复项
- 改进名称
- 提取助手
- 简化表达

始终保持测试绿色。不要添加行为。

**如果重构期间测试失败：** 立即撤消。采取较小的步骤。

### 重复

下一个行为的下一个失败测试。一次一个循环。

## 为什么顺序很重要

**“我会在之后编写测试来验证它是否有效”**

代码通过后编写的测试立即通过。立即通过并不能证明什么：
- 可能会测试错误的东西
- 可能测试实施，而不是行为
- 可能会错过你忘记的边缘情况
- 你从未见过它捕获错误

测试优先迫使您看到测试失败，证明它确实测试了某些东西。

**“我已经手动测试了所有边缘情况”**

手动测试是临时的。您认为您测试了所有内容，但：
- 没有您测试内容的记录
- 代码更改时无法重新运行
- 在压力下容易忘记案件
- “当我尝试它时它起作用了”≠全面

自动化测试是系统化的。他们每次都以同样的方式奔跑。

**“删除 X 小时的工作是浪费”**

沉没成本谬误。时间已经过去了。您现在的选择：
- 使用 TDD 删除并重写（高可信度）
- 保留它并在之后添加测试（低置信度，可能存在错误）

“浪费”是保留你不信任的代码。

**“TDD 是教条主义的，务实意味着适应”**

TDD 很务实：
- 在提交之前查找错误（比之后调试更快）
- 防止回归（测试立即捕获中断）
- 文档行为（测试显示如何使用代码）
- 启用重构（自由更改，测试捕获中断）

“实用”快捷方式=生产中的调试=速度较慢。

**“达到相同目标后进行测试——这是精神而不是仪式”**

不。回答后测试“这有什么作用？”测试优先回答“这应该做什么？”

之后的测试因您的实现而存在偏差。您测试您构建的内容，而不是测试所需的内容。测试优先强制在实施之前发现边缘情况。

## 常见的合理化

|对不起|现实|
|--------|---------|
| “太简单了，无法测试” |简单的代码中断。测试需要 30 秒。 |
| “之后我会测试” |测试立即通过并不能证明什么。 |
| “达到相同目标后进行测试” | Tests-after =“这是做什么的？”测试优先=“这应该做什么？” |
| “已经手动测试”|临时性≠系统性。没有记录，无法重新运行。 |
| “删除 X 小时是浪费”|沉没成本谬误。保留未经验证的代码是技术债。 |
| “留作参考，先写测试” |你会适应它。后面就是测试了删除就是删除的意思。 |
| “需要先探索” |美好的。放弃探索，从 TDD 开始。 |
| “努力测试=设计不清楚” |听听测试。难以测试=难以使用。 |
| “TDD 会让我放慢速度” | TDD 比调试更快。务实=测试第一。 |
| “手动测试速度更快”|手册不能证明边缘情况。您将重新测试每个更改。 |
| “现有代码没有经过测试”|你正在改进它。为您接触的代码添加测试。 |

## 危险信号 — 停止并重新开始

如果您发现自己正在执行其中任何操作，请删除代码并使用 TDD 重新启动：

- 测试前的代码
- 实施后测试
- 第一次运行时测试立即通过
- 无法解释测试失败的原因
-“稍后”添加的测试
- 合理化“就这一次”
- “我已经手动测试过了”
- “达到相同目的后进行测试”
- “保留作为参考”或“改编现有代码”
- “已经花了X小时，删除是浪费”
- “TDD 很教条，我很务实”
- “这是不同的，因为......”

**所有这些意味着：删除代码。从 TDD 开始。**

## 验证清单

在标记工作完成之前：

- [ ] 每个新函数/方法都有一个测试
- [ ] 在实施之前观察每个测试的失败
- [ ] 每个测试均因预期原因而失败（功能缺失，而非拼写错误）
- [ ] 编写最少的代码来通过每个测试
- [ ] 所有测试均通过
- [ ] 输出原始（没有错误、警告）
- [ ] 测试使用真实代码（仅在不可避免时才进行模拟）
- [ ] 涵盖边缘情况和错误

无法选中所有框吗？你跳过了 TDD。重新开始。

## 当卡住时

|问题 |解决方案 |
|---------|----------|
|不知道如何测试 |编写所需的 API。先写断言。询问用户。 |
|测试太复杂|设计太复杂了。简化界面。 |
|必须嘲笑一切|代码耦合性太强。使用依赖注入。 |
|测试设置巨大|提取助手。还是很复杂？简化设计。 |

## OpenClaw 代理集成

### 运行测试

使用“terminal”工具在每一步运行测试：

````蟒蛇
# 红色 — 验证失败
终端（“pytest测试/test_feature.py::test_name -v”）

# 绿色 — 验证通过
终端（“pytest测试/test_feature.py::test_name -v”）

# 全套 — 验证没有回归
终端（“pytest 测试/-q”）
````

### 使用 delegate_task

当调度子代理进行实施时，在目标中强制执行 TDD：

````蟒蛇
委托任务（
    goal="使用严格的 TDD 实现 [功能]",
    上下文=“”“
    遵循测试驱动开发技能：
    1. 首先编写失败的测试
    2. 运行测试以验证是否失败
    3. 编写最少的代码即可通过
    4. 运行测试以验证其通过
    5. 如果需要的话重构
    6. 承诺

    项目测试命令：pytesttests/-q
    项目结构：[描述相关文件]
    """,
    工具集=['终端', '文件']
）
````

### 通过系统调试

发现错误了吗？编写失败的测试来重现它。遵循 TDD 周期。测试证明了修复并防止了回归。

未经测试切勿修复错误。

## 测试反模式

- **测试模拟行为而不是真实行为** - 模拟应该验证交互，而不是替换被测系统
- **测试实现细节** - 测试行为/结果，而不是内部方法调用
- **仅快乐之路** - 始终测试边缘情况、错误和边界
- **脆性测试** — 测试应该验证行为，而不是结构；重构不应该破坏它们

## 最终规则

````
生产代码→测试存在并且首先失败
否则 → 不是 TDD
````

未经用户明确许可，无一例外。
