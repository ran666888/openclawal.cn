# 子代理驱动的开发

## 概述

通过为每项任务派遣新的子代理并进行系统的两阶段审查来执行实施计划。

**核心原则：** 每个任务新鲜的子代理+两阶段审查（规格然后质量）=高质量，快速迭代。

## 何时使用

在以下情况下使用此技能：
- 您有一个实施计划（来自“计划”技能或用户需求）
- 任务大多是独立的
- 质量和规格合规性很重要
- 您希望在任务之间进行自动审查

**与。手动执行：**
- 每个任务的新上下文（不会因累积状态而混淆）
- 自动审查流程尽早发现问题
- 所有任务的一致质量检查
- 子代理可以在开始工作前提问

## 过程

### 1. 阅读并解析计划

阅读计划文件。预先提取所有任务及其全文和上下文。创建一个待办事项列表：

````蟒蛇
# 阅读计划
read_file(“文档/计划/feature-plan.md”)

# 创建包含所有任务的待办事项列表
待办事项（[
    {"id": "task-1", "content": "使用电子邮件字段创建用户模型", "status": "pending"},
    {"id": "task-2", "content": "添加密码哈希实用程序", "status": "pending"},
    {"id": "task-3", "content": "创建登录端点", "status": "pending"},
]）
````

**关键：** 阅读该计划一次。提取所有内容。不要让子代理读取计划文件 - 直接在上下文中提供完整的任务文本。

### 2. 每个任务的工作流程

对于计划中的每项任务：

#### 第 1 步：调度实施者子代理

使用具有完整上下文的“delegate_task”：

````蟒蛇
委托任务（
    goal="实施任务 1：使用电子邮件和密码哈希字段创建用户模型",
    上下文=“”“
    计划任务：
    - 创建：src/models/user.py
    - 添加带有电子邮件（str）和密码哈希（str）字段的用户类
    - 使用bcrypt进行密码散列
    - 包含 __repr__ 用于调试

    关注 TDD：
    1. 在tests/models/test_user.py中编写失败的测试
    2.运行：pytest测试/模型/test_user.py -v（验证失败）
    3. 编写最小实现
    4.运行：pytest测试/模型/test_user.py -v（验证通过）
    5.运行：pytest测试/-q（验证没有回归）
    6. 提交： git add -A && git commit -m "feat: 添加带有密码哈希的用户模型"

    项目背景：
    - Python 3.11，Flask 应用程序位于 src/app.py
    - src/models/ 中的现有模型
    - 测试使用 pytest，从项目根目录运行
    - bcrypt 已在requirements.txt中
    """,
    工具集=['终端', '文件']
）
````

#### 第 2 步：派遣规范合规审核员

实施者完成后，根据原始规范进行验证：

````蟒蛇
委托任务（
    goal="审查实施是否符合计划中的规范",
    上下文=“”“
    原始任务规范：
    - 使用 User 类创建 src/models/user.py
    - 字段：电子邮件 (str)、password_hash (str)
    - 使用bcrypt进行密码散列
    - 包括 __repr__

    检查：
    - [ ] 规范中的所有要求均已实施吗？
    - [ ] 文件路径符合规范吗？
    - [ ] 函数签名与规范相符吗？
    - [ ] 行为符合预期吗？
    - [ ] 没有添加任何额外内容（无范围蔓延）？

    输出：通过或需要修复的特定规格差距列表。
    """,
    工具集=['文件']
）
````

**如果发现规格问题：** 修复差距，然后重新运行规格审核。仅当符合规范时才继续。

#### 第 3 步：派遣代码质量审核员

规范合规性通过后：

````蟒蛇
委托任务（
    goal="审查任务 1 实施的代码质量",
    上下文=“”“
    要审查的文件：
    - src/models/user.py
    - 测试/模型/test_user.py

    检查：
    - [ ] 遵循项目惯例和风格？
    - [ ] 正确的错误处理？
    - [ ] 清除变量/函数名称？
    - [ ] 测试覆盖率足够吗？
    - [ ] 没有明显的错误或遗漏的边缘情况？
    - [ ] 没有安全问题？

    输出格式：
    - 关键问题：[必须在继续之前解决]
    - 重要问题：[应该修复]
    - 小问题：[可选]
    - 判决：已批准或 REQUEST_CHANGES
    """,
    工具集=['文件']
）
````

**如果发现质量问题：** 修复问题，重新审核。仅在获得批准后才继续。

#### 第 4 步：标记为完成

````蟒蛇
todo([{"id": "task-1", "content": "使用电子邮件字段创建用户模型", "status": "已完成"}], merge=True)
````

### 3. 最终审核

所有任务完成后，派遣最终集成审核员：

````蟒蛇
委托任务（
    goal="审查整个实施的一致性和集成问题",
    上下文=“”“
    计划中的所有任务均已完成。查看完整实施情况：
    - 所有组件都可以一起工作吗？
    - 任务之间是否存在不一致？
    - 所有测试都通过了吗？
    - 准备好合并了吗？
    """,
    工具集=['终端', '文件']
）
````

### 4.验证并提交

````bash
# 运行完整的测试套件
pytest 测试/-q

# 查看所有更改
git diff --stat

# 如果需要的话最后提交
git add -A && git commit -m "feat: 完成 [功能名称] 实现"
````

## 任务粒度

**每项任务 = 2-5 分钟的专注工作。**

**太大：**
- “实施用户认证系统”

**合适的尺寸：**
- “使用电子邮件和密码字段创建用户模型”
- “添加密码哈希函数”
- “创建登录端点”
- “添加 JWT 令牌生成”
- “创建注册端点”

## 危险信号 — 切勿执行这些操作

- 没有计划就开始实施
- 跳过审查（规范合规性或代码质量）
- 继续处理未解决的关键/重要问题
- 为涉及相同文件的任务调度多个实施子代理
- 让子代理读取计划文件（在上下文中提供全文）
- 跳过场景设置上下文（子代理需要了解任务适合的位置）
- 忽略子代理问题（在继续之前回答）
- 接受规范合规性“足够接近”
- 跳过审核循环（审核者发现问题→实施者修复→再次审核）
- 让实施者自我审查取代实际审查（两者都需要）
- **在规范合规性通过之前开始代码质量审查**（错误顺序）
- 当任一审核有未决问题时移至下一个任务

## 处理问题

### 如果子代理询问问题

- 回答清楚、完整
- 如果需要，提供额外的背景信息
- 不要急于实施

### 如果审阅者发现问题

- 实施者子代理（或新的子代理）修复了这些问题
- 审稿人再次审稿
- 重复直至获得批准
- 不要跳过重新审核

### 如果子代理任务失败

- 派遣一个新的修复子代理，并提供有关问题所在的具体说明
- 不要尝试在控制器会话中手动修复（上下文污染）

## 效率说明

**为什么每个任务需要新鲜的子代理：**
- 防止累积状态造成上下文污染
- 每个子代理都获得干净、集中的上下文
- 不会与先前任务的代码或推理混淆

**为什么要进行两阶段审查：**
- 规范审查尽早发现建设不足/过度建设
- 质量审查确保实施良好
- 在问题跨任务复合之前发现问题

**成本权衡：**
- 更多子代理调用（每个任务的实施者 + 2 名审阅者）
- 但尽早发现问题（比稍后调试复合问题便宜）

## 与其他技能的整合

### 有计划

该技能执行由“plan”技能创建的计划：
1、用户需求→计划→实施计划
2. 实施方案→子代理驱动开发→工作代码

### 通过测试驱动开发

实施者子代理应遵循 TDD：
1. 首先编写失败的测试
2. 实现最少的代码
3. 验证测试通过
4. 承诺

在每个实施者上下文中包含 TDD 指令。

### 请求代码审查

两阶段审查过程就是代码审查。对于最终集成审核，请使用请求代码审核技能的审核维度。

### 通过系统调试

如果子代理在实施过程中遇到错误：
1.遵循系统调试流程
2. 修复前查找根本原因
3.编写回归测试
4. 恢复实施

## 工作流程示例

````
[阅读计划：docs/plans/auth-feature.md]
[创建包含 5 个任务的待办事项列表]

--- 任务 1：创建用户模型 ---
[调度实施者子代理]
  实施者：“电子邮件应该是唯一的吗？”
  您：“是的，电子邮件必须是唯一的”
  实施者：已实施，3/3 测试通过，已承诺。

[派遣规格评审员]
  规格审核员：✅ 通过 — 满足所有要求

【派遣质量评审员】
  质量审核员：✅ 已批准 — 干净的代码，良好的测试

[标记任务 1 已完成]

--- 任务 2：密码哈希 ---
[调度实施者子代理]
  实施者：没有问题，已实施，5/5 测试通过。

[派遣规格评审员]
  规范审核者：❌ 缺少：密码强度验证（规范说“最少 8 个字符”）

[实施者修复]
  实施者：添加了验证，7/7 测试通过。

[再次派遣规格评审员]
  规格审核员： ✅ 通过

【派遣质量评审员】
  质量审核员：重要：神奇数字 8，提取为常数
  实施者：提取的 MIN_PASSWORD_LENGTH 常量
  质量审核员： ✅ 已批准

[标记任务2已完成]

...（继续执行所有任务）

[全部任务结束后：派遣最终整合审核员]
[运行完整的测试套件：全部通过]
[完成！]
````

## 记住

````
每个任务都有新的子代理
每次都分两阶段审核
规范合规性第一
代码质量第二
永远不要跳过评论
尽早发现问题
````

**质量并非偶然。这是系统化过程的结果。**
