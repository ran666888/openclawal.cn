# 对抗性用户体验测试

扮演产品中最坏情况的用户——讨厌技术、不想要你的软件、并且会找到各种理由抱怨的人。然后通过实用主义层过滤他们的反馈，将真正的用户体验问题与“我讨厌计算机”的噪音分开。

把它想象成一个自动化的“妈妈测试”——但很生气。

## 为什么这有效

大多数 QA 都会发现错误。这会发现**摩擦**。技术上正确的应用程序仍然可能无法供真人使用。敌对角色抓住了：
- 令人困惑的术语，对开发人员有意义，但对用户无意义
- 完成基本任务的步骤太多
- 缺少入职或“顿悟时刻”
- 辅助功能问题（字体大小、对比度、点击目标）
- 冷启动问题（空状态，无演示内容）
- 付费墙/注册摩擦会影响转化

**实用主义过滤器**（第 3 阶段）使其变得有用而不仅仅是娱乐。如果没有它，您将在每个屏幕上添加一个“打印此页”按钮，因为爷爷无法识别 PDF。

## 如何使用

告诉代理：
````
“在 [URL] 上运行对抗性 UX 测试”
“做一个脾气暴躁的[角色类型]并测试[应用程序名称]”
“在我的临时网站上进行混蛋用户测试”
````

您可以提供一个角色，或者让代理根据您产品的目标受众生成一个角色。

## 步骤 1：定义角色

如果未提供角色，请通过回答以下内容生成一个角色：

1. **谁是该产品最难使用的用户？**（50 岁以上，非技术角色，数十年“老方法”使用经验）
2. **他们的技术舒适度是多少？**（越低越好——仅限 WhatsApp、纸质笔记本、妻子设置了电子邮件）
3. **他们需要完成的一件事是什么？**（他们的核心工作，而不是你的功能列表）
4. **什么会让他们放弃？**（点击太多、行话、速度慢、令人困惑）
5. **他们在沮丧时如何说话？**（直率、咒骂、不屑一顾、叹息）

### 良好的角色示例
> **“Big Mick”McAllister** — 58 岁的 S&C 教练。使用 WhatsApp 就这样了。他的“电子表格”是纸质笔记本。 “如果我不能在 10 秒内弄清楚，我就会回到笔记本上。”需要记录 25 名玩家的会话结果。讨厌小文本、行话和密码。

### 不良角色示例
> “不喜欢该应用程序的用户”——太模糊，没有限制，没有声音。

角色必须**足够具体，才能在 20 分钟的测试中保持角色**。

## 步骤 2：成为混蛋（作为角色浏览）

1. 阅读任何可用的项目文档以了解应用程序上下文和 URL
2. **完全融入角色**——他们的挫败感、局限性、目标
3. 使用浏览器工具导航到应用程序
4. **尝试角色的实际任务**（不是专题导览）：
   - 他们能做他们想做的事吗？
   - 需要多少次点击/屏幕才能完成？
   - 是什么让他们感到困惑？
   - 是什么让他们生气？
   - 他们在哪里迷路？
   - 什么会让他们放弃并回到原来的方式？

5. 测试这些摩擦类别：
   - **第一印象** - 他们甚至会费心浏览登陆页面吗？
   - **核心工作流程** — 他们最常需要做的一件事
   - **错误恢复** — 当他们做错事时会发生什么？
   - **可读性** — 文本大小、对比度、信息密度
   - **速度** — 感觉比他们当前的方法更快吗？
   - **术语** — 有他们不懂的行话吗？
   - **导航** — 他们能找到回去的路吗？他们知道自己在哪里吗？

6.对每个痛点进行截图
7. 检查浏览器控制台每个页面是否有 JS 错误

## 步骤 3：咆哮（用角色写反馈）

将反馈写成人物角色——用他们的声音，带着他们的挫败感。这不是错误报告。这才是真正的人类发泄。

````
[人物名称] 对 [产品] 的评论

总体而言：[他们会继续使用它吗？是/否/也许有条件]

好的（勉强承认）：
——【连他们也不得不承认的事情】

不好的（合法的用户体验问题）：
- [会阻止他们使用该产品的实际问题]

丑陋的（搅局者）：
- [会让他们立即卸载/取消的事情]

具体投诉：
1. [页面/专题]：“[人物语音引述]” — [发生了什么，预期]
2....

结论：“[一句话总结他们的经历]”
````

## 步骤 4：实用主义过滤器（关键 — 请勿跳过）

走出角色。作为产品人员评估每个投诉：

- **红色：真正的用户体验错误** — 任何用户都会遇到这个问题，而不仅仅是脾气暴躁的用户。修复它。
- **黄色：有效但优先级低** — 真正的问题，但仅适用于极端用户。注意它。
- **白色：人格噪音** — “我讨厌电脑”的说法，而不是产品问题。跳过它。
- **绿色：功能请求** — 隐藏在投诉中的好主意。考虑一下。

### 过滤标准
1. 一位 35 岁、有能力但忙碌的用户也会有同样的抱怨吗？ → 红色
2. 这是一个真正的可访问性问题（字体大小、对比度、点击目标）吗？ → 红色
3. 这种“我希望它像纸一样工作”是对数字化的抵制吗？ → 白色
4. 这是角色偶然发现的真正工作流程效率低下的情况吗？ → 黄色或红色
5. 对于 80% 的人来说，解决这个问题会增加复杂性吗？ → 白色
6. 投诉是否表明缺少入职时刻？ → 绿色

**此过滤器是强制性的。**切勿将原始角色投诉作为票证发送。

## 步骤 5：创建票证

仅适用于 **红色** 和 **绿色** 商品：
- 清晰、可操作的标题
- 包括角色的逐字引用（有趣+难忘）
- 真正的用户体验问题（客观）
- 建议的修复（可操作）
- 标签/标签：“ux-review”

对于 **黄色** 物品：一张包含所有注释的全能票。

**白色**项目仅出现在报告中。没有门票。

**每节最多 10 张票** — 专注于最糟糕的问题。

## 第 6 步：报告

交付：
1. 角色咆哮（第 3 步）——有趣且发自内心
2. 经过筛选的评估（步骤 4）——务实且可操作
3. 创建票证（第 5 步）——带有链接
4、关键问题截图

## 提示

- **每次会话一个角色。** 不要混合观点。
- **在步骤 2-3 中保持角色。** 仅在步骤 4 中打破角色。
- **首先测试核心工作流程。** 不要被设置页面分散注意力。
- **空状态是金。** 新的用户体验揭示了最大的摩擦。
- **最好的发现是角色在尝试做其他事情时意外发现的红色物品**。
- **如果角色的投诉为零，那么你的角色就太精通技术了。** 让他们变老，缺乏耐心，更加固守自己的方式。
- **在演示、发布之前或发布一批功能之后运行此命令。**
- **尽可能注册为新用户。** 不要使用预先播种的管理员帐户 - 冷启动体验是最有摩擦的地方。
- **零白色物品是一个信号，而不是失败。** 如果实用主义过滤器没有发现噪音，那么你的产品就有真正的用户体验问题，而不仅仅是一个脾气暴躁的角色。
- **测试后检查项目文档中的已知问题。** 如果角色发现已知问题列表中已有的错误，这实际上是最糟糕的发现 - 这意味着团队知道它，但从未感受到用户的痛苦。
- **订阅/付费墙测试至关重要。** 使用过期帐户进行测试，而不仅仅是活动帐户。 “当你无法支付时会发生什么”体验揭示了产品是否尊重用户或劫持他们的数据。
- **计算完成角色的一项任务的点击次数。** 如果超过 5 次，则无论角色技术水平如何，这几乎总是红色结果。

## 规则

- 在第 2-3 步中保持本色
- 真诚而公平——发现真正的问题，而不是人为制造的问题
- 实用主义过滤器（步骤 4）是**强制**
- 每项投诉都需要截图
- 每场最多 10 张门票
- 在登台/部署的应用程序上进行测试，而不是在本地开发上进行测试
- 一个角色，一场会议，一份报告
