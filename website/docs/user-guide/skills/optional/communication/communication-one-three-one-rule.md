# 一三一规则

用于技术建议和权衡分析的结构化决策框架。当用户面临多种方法（架构决策、工具选择、重构策略、迁移路径）之间的选择时，该技能会产生 1-3-1 格式：一个清晰的问题陈述、三个具有优点/缺点的不同选项，以及一个包含已完成定义和实施计划的具体建议。当用户要求“1-3-1”、说“给我选项”或需要帮助在竞争方法之间进行选择时使用。

#1-3-1 通讯规则

当任务有多种可行方法并且用户需要明确的建议时，采用结构化决策格式。生成一个简洁的问题框架、三个权衡选项以及针对推荐路径的可行计划。

## 何时使用

- 用户明确要求“1-3-1”响应。
- 用户对技术决策说“给我选项”或“我的选择是什么”。
- 一项任务具有多种可行的方法，并进行有意义的权衡（架构、工具、迁移策略）。
- 用户需要一个可以转发给团队或利益相关者的提案。

请勿用于具有明显答案的简单问题、调试会话或用户已决定一种方法的任务。

## 程序

1. **问题**（一句话）
   - 用简洁的句子陈述核心决定或期望的结果。
   - 关注“什么”，而不是“如何”——没有实现细节、工具名称或特定技术。
   - 保持紧。如果您需要“和”，那么您正在描述两个问题。

2. **选项**（正好三个）
   - 提出三种不同的、可行的方法，标记为 A、B、C。
   - 每个选项都有简短的描述、优点和缺点。
   - 选项应该代表真正不同的策略，而不是同一方法的微小变化。

3. **推荐**（一项）
   - 根据用户的背景和优先级，说明您推荐的选项及其原因。
   - 直接一点——这是你的专业判断，而不是对冲。

4. **完成的定义**
   - 列出推荐选项的具体成功标准。
   - 这些是具体的、可验证的结果——而不是模糊的愿望。
   - 如果用户选择不同的选项，请修改此部分以匹配。

5. **实施计划**
   - 执行推荐选项的具体步骤。
   - 包括适用的特定命令、工具或操作。
   - 如果用户选择不同的选项，请修改此部分以匹配。

## 验证

- 响应只有一个问题句。
- 响应正好有三个选项（A、B、C），每个选项都有优点和缺点。
- 响应有一个建议，通过推理选择一个选项。
- 完成的定义和实施计划与建议的选项一致。
- 如果用户选择不同的选项，建议、DoD 和实施计划也会相应更新。

## 示例

用户：“给我一个 1-3-1，以便向我们的 API 客户端添加重试逻辑。”

代理：

**问题：** 对外部服务的 API 调用在负载下间歇性失败，需要优雅的重试处理。

**选项：**

- **选项 A：共享实用程序中带有抖动的指数退避。**
  优点：简单、易于理解的模式；所有调用者共享单一实现。
  缺点：不能断路；调用者在重试期间仍然会阻塞。

- **选项 B：在每个端点调用上重试库（坚韧）作为装饰器。**
  优点：每个端点均可配置；对断路器和速率限制的内置支持。
  缺点：增加依赖；装饰器模式可能会掩盖控制流。

- **选项 C：基于队列的异步重试和死信处理。**
  优点：非阻塞；失败的呼叫会被跟踪并可重播；扩展性良好。
  缺点：复杂性显着增加；需要一个队列后端；对于小批量服务来说过度杀伤力。

**建议：** 选项 A。调用量并不能证明队列基础设施的合理性，并且具有指数退避 + 抖动的共享实用程序可以干净地覆盖故障模式，而无需新的依赖项。

**完成的定义：**
- `api_retry()` 实用程序处理瞬时 HTTP 错误（429、502、503、504）。
- 重试最多 3 次，并具有指数退避和抖动。
- 所有外部 API 调用都通过重试实用程序进行路由。
- 记录每次重试尝试以及端点和尝试编号。
- 测试涵盖重试耗尽、立即成功和暂时成功场景。

**实施计划：**
1. 创建具有可配置最大重试次数、基本延迟和可重试状态代码的 `utils/api_retry.py`。
2. 使用 random.uniform(0, base_delay) 添加抖动以防止雷群。
3. 使用重试实用程序将现有 API 调用包装在“api_client.py”中。
4. 为每个重试场景添加模拟 HTTP 响应的单元测试。
5. 通过针对片状端点模拟的简单压力测试来验证负载情况。
