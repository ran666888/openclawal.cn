# 工具搜索

当您有许多 MCP 服务器或附加到一个非核心插件工具时
会话，它们的 JSON 模式可以消耗很大一部分
每个回合的上下文窗口——即使只有其中几个是相关的
用户实际要求的内容。

**工具搜索**是 OpenClaw 的选择加入渐进披露层
问题。激活后，MCP 和插件工具将被替换为
模型可见工具由三个桥接工具组成，模型加载每个工具
按需特定工具的架构。

:::info 内置 OpenClaw 工具永不延迟
构成 OpenClaw 核心功能集的工具（“终端”、
`读文件`、`写文件`、`补丁`、`搜索文件`、`todo`、`内存`、
`browser_*`、`web_search`、`web_extract`、`clarify`、`execute_code`、
`delegate_task`、`session_search`、`send_message` 以及其余的
`_HERMES_CORE_TOOLS`）*总是*直接加载。仅 MCP 工具和
非核心插件工具有资格延期。
:::

## 它是如何工作的

当工具搜索激活一圈时，模型会在其中看到三个新工具
延期地点：

````
tool_search(query, limit?) — 搜索延迟工具目录
tool_describe(name) — 加载一个工具的完整模式
tool_call(name,arguments) — 调用延迟工具
````

典型的交互如下所示：

````
模型：tool_search(“创建 github 问题”)
  → { 匹配：[{ name: "mcp_github_create_issue", ... }, ...] }
模型：tool_describe(“mcp_github_create_issue”)
  → { 参数：{ 类型：“对象”，属性：{ ... } } }
模型：tool_call("mcp_github_create_issue", { 标题: "...", 正文: "..." })
  → { 好的：true，问题编号：42 }
````

当模型调用“tool_call”时，OpenClaw **解开桥**并
完全像模型调用它一样调度底层工具
直接。工具调用前挂钩、护栏、批准提示和
工具调用后挂钩全部针对真实工具名称运行 - 而不是针对
`工具调用`。 CLI 和网关中的活动源也会解开，以便您
查看底层工具，而不是桥梁。

## 什么时候激活？

默认情况下，工具搜索以“自动”模式运行：仅当
可延迟工具模式将消耗至少 10% 的活动模型
上下文窗口。在此之下，工具阵列组件是一个纯粹的
直通，您无需支付任何管理费用。

每次构建工具数组时都会重新评估此决定，因此：

- 只使用几个 MCP 工具和一个长上下文模型的会话永远不会
  激活工具搜索。
- 连接许多 MCP 服务器（通常超过 15 个工具）的会话启动
  激活它。
- 在会话中删除MCP服务器可以正确返回到直接暴露
  在下一次大会上。

## 配置

````yaml
工具：
  工具搜索：
    enabled: auto # 自动（默认）、打开或关闭
    Threshold_pct: 10 # 上下文百分比 — 仅在自动模式下使用
    搜索默认限制：5
    最大搜索限制：20
````

|关键|默认 |意义|
| --- | --- | --- |
| `已启用` | `自动` | `auto` 在阈值以上激活；如果至少有一个可延迟工具，则“on”始终激活； “off”完全禁用。 |
| `threshold_pct` | `10` | “自动”模式启动时上下文长度的百分比。范围 0–100。 |
| `搜索默认限制` | `5` |当模型调用没有“limit”的“tool_search”时返回命中。 |
| `最大搜索限制` | `20` |模型可以通过“limit”请求的硬上限。范围 1–50。 |

您还可以翻转传统的布尔形状：

````yaml
工具：
  tool_search: true # 相当于 {enabled: auto}
````

## 何时不使用它

工具搜索交易固定的每回合代币成本（三桥工具
模式，~300 个令牌）和至少一次额外的往返（搜索 →
描述 → 调用）以节省延迟模式。这是一个明确的
当你拥有很多工具并且每回合使用很少的工具时就会获胜；这是开销，当
你总共只有很少的工具。

“auto”默认值会为您处理这个问题。如果您设置 `enabled: on`
无条件地，预计小型工具集的每转成本会略有增加。

## 不会消失的权衡

这些来自提示缓存完整性不变量——它们是固有的
对于任何渐进式披露设计，不特定于此实现：

- **在冷工具上进行一次额外的往返。**模型第一次需要
  一个延迟工具，它花费一两个额外的模型调用来查找和
  加载架构。静态方面的代币节省是真实的，但是
  部分在运行时偿还。
- **延迟模式没有缓存优势。**加载的“tool_describe”
  结果进入对话历史记录（因此它确实被缓存在
  后续回合）但它永远不会从系统提示缓存中受益
  前缀。
- **模型质量依赖性。**工具搜索假设模型可以编写
  对其想要的工具进行合理的搜索查询。较小的型号可以这样做
  不太好；已发表的人择数（Opus 4 上为 49% → 74%）
  与没有工具搜索的情况相比）显示了优点，但也显示了约 26 点
  准确度仍然是检索失败。
- **工具集编辑使缓存无效。** 添加或删除工具
  会话更改桥接工具的描述（其中包括
  延迟工具的计数）和目录，因此提示缓存是
  无效。这与任何工具集编辑都是相同的权衡。

## 实施细节

- **检索：** BM25 over tokenized 工具名称 + 描述 + 参数
  名称。当以下情况时，回退到工具名称的字面子字符串匹配
  BM25 不会返回正分数命中，这可以防止
  零 IDF 退化情况（例如，针对 a 搜索“github”）
  每个工具名称都包含“github”的目录）。
- **目录在各个轮次中是无状态的。**它从当前的目录重建
  tool-defs 列出了每个程序集 — 没有会话键控的“Map”。这样就避免了
  存储的目录与系统不同步的错误类别
  实时工具注册表。
- **目录的范围仅限于会话的工具集。** `tool_search`,
  `tool_describe` 和 `tool_call` 只能看到并调用工具
  会议实际上已获得批准。子代理、看板工作人员或网关
  仅限于工具集子集的会话无法使用桥接
  发现或调用该子集之外的工具 - 延迟目录是
  会话自己的启用/禁用工具集的可延迟部分，
  不是整个进程注册表。
- **无 JS 沙箱。** OpenClaw 使用更简单的“结构化工具”模式
  （搜索/描述/调用为普通函数）。 JS沙盒”代码
  模式”，其他一些实现提供的是大表面积；我们
  跳过它。
