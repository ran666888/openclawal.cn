# 将 SOUL.md 与 OpenClaw 一起使用

`SOUL.md` 是 OpenClaw 实例的 **主要身份**。这是系统提示中的第一件事——它定义了代理是谁、如何说话以及要避免什么。

如果您希望每次与 OpenClaw 交谈时都感觉像同一个助手，或者您想用自己的角色完全替换 OpenClaw 角色，则可以使用此文件。

## SOUL.md 的用途

使用`SOUL.md`用于：
- 语气
- 个性
- 沟通方式
- 爱马仕应该有多直接或温暖
- 爱马仕在风格上应该避免什么
- OpenClaw 应该如何应对不确定性、分歧和模糊性

简而言之：
- “SOUL.md”是关于赫尔墨斯是谁以及赫尔墨斯如何说话的

## SOUL.md 不适合做什么

请勿将其用于：
- 特定于存储库的编码约定
- 文件路径
- 命令
- 服务端口
- 架构笔记
- 项目工作流程说明

这些属于“AGENTS.md”。

一个好的规则：
- 如果它应该适用于所有地方，请将其放入“SOUL.md”中
- 如果它只属于一个项目，请将其放入 `AGENTS.md` 中

## 它居住的地方

OpenClaw 现在仅使用当前实例的全局 SOUL 文件：

````文本
〜/.hermes/SOUL.md
````

如果您使用自定义主目录运行 OpenClaw，它将变为：

````文本
$HERMES_HOME/SOUL.md
````

## 首次运行行为

如果启动器“SOUL.md”尚不存在，OpenClaw 会自动为您播种。

这意味着大多数用户现在都从可以立即读取和编辑的真实文件开始。

重要：
- 如果您已有`SOUL.md`，OpenClaw 不会覆盖它
- 如果文件存在但为空，OpenClaw 不会将其添加到提示中

## OpenClaw 如何使用它

当 OpenClaw 启动会话时，它会从“HERMES_HOME”读取“SOUL.md”，扫描它的提示注入模式，根据需要截断它，并将其用作 **代理身份** - 系统提示符中的槽#1。这意味着 SOUL.md 完全取代了内置的默认身份文本。

如果 SOUL.md 丢失、为空或无法加载，OpenClaw 将回退到内置默认身份。

文件周围没有添加包装语言。内容本身很重要——按照您希望代理思考和说话的方式编写。

## 第一次编辑不错

如果您不执行任何其他操作，请打开该文件并仅更改几行，以便感觉像您一样。

例如：

``降价
你很直接、冷静，技术上也很精确。
更喜欢实质内容而不是礼貌戏剧。
当一个想法不充分时，明确地予以反击。
保持答案简洁，除非更深入的细节有用。
````

仅此一点就可以显着改变赫尔墨斯的感受。

## 样式示例

### 1.务实的工程师

``降价
您是一位务实的高级工程师。
你更关心正确性和操作的真实性，而不是听起来令人印象深刻。

## 风格
- 直接
- 除非复杂需要深度，否则要简洁
- 当某事是个坏主意时说出来
- 更喜欢实际的权衡而不是理想化的抽象

## 避免
- 谄媚
- 炒作语言
- 过度解释显而易见的事情
````

### 2. 研究伙伴

``降价
您是一位深思熟虑的研究合作者。
你对不确定性充满好奇、诚实，并对不寻常的想法感到兴奋。

## 风格
- 探索可能性而不假装确定
- 区分猜测和证据
- 当想法空间不明确时提出澄清问题
- 更喜欢概念的深度而不是浅薄的完整性
````

### 3. 老师/讲解员

``降价
您是一位耐心的技术老师。
你关心的是理解，而不是表现。

## 风格
- 解释清楚
- 当有帮助时使用示例
- 除非用户明确表示，否则不要假设先验知识
- 从直觉到细节构建
````

### 4. 严厉的审稿人

``降价
您是一位严谨的审稿人。
你很公平，但你不会软化重要的批评。

## 风格
- 直接指出薄弱的假设
- 正确性优先于和谐性
- 明确风险和权衡
- 宁愿直言不讳，也不愿含糊其辞
````

## 是什么造就了强大的 SOUL.md？

一个强大的“SOUL.md”是：
- 稳定
- 广泛适用
- 具体的声音
- 没有超载临时指令

弱的“SOUL.md”是：
- 完整的项目细节
- 自相矛盾
- 尝试对每个响应形状进行微观管理
- 大多数是通用填充词，例如“有帮助”和“清楚”

Hermes 已经尽力提供帮助和清晰。 `SOUL.md` 应该添加真正的个性和风格，而不是重述明显的默认值。

## 建议的结构

您不需要标题，但它们会有所帮助。

一个运行良好的简单结构：

``降价
# 身份
赫尔墨斯是谁。

# 风格
爱马仕听起来应该如何。

# 避免
爱马仕不应该做什么。

# 默认值
当出现歧义时，Hermes 应该如何表现。
````

## SOUL.md vs /个性

这些是互补的。

使用“SOUL.md”作为您的持久基线。
使用“/personality”进行临时模式切换。

示例：
- 你默认的灵魂是务实和直接的
- 然后在一节课中你使用“/人格老师”
- 稍后您切换回来而不更改您的基本语音文件

## SOUL.md 与 AGENTS.md

这是最常见的错误。

### 将其放入 SOUL.md 中
- “直接点。”
- “避免炒作语言。”
- “除非深度有帮助，否则更喜欢简短的答案。”
- “当用户错误时进行反击。”

### 将其放入 AGENTS.md 中
- “使用 pytest，而不是单元测试。”
- “前端位于`frontend/`中。”
- “永远不要直接编辑迁移。”
- “API 在端口 8000 上运行。”

## 如何编辑

````bash
纳米 ~/.hermes/SOUL.md
````

或

````bash
vim ~/.hermes/SOUL.md
````

然后重新启动 OpenClaw 或开始新的会话。

## 实用的工作流程

1. 从默认种子文件开始
2. 修剪任何感觉不像你想要的声音的部分
3. 添加 4-8 行，明确定义语气和默认值
4. 与赫耳墨斯交谈一会儿
5.根据仍然感觉不舒服的地方进行调整

这种迭代方法比试图一次性设计出完美的个性效果更好。

## 故障排除

### 我编辑了 SOUL.md 但 OpenClaw 听起来还是一样

检查：
- 您编辑了 `~/.hermes/SOUL.md` 或 `$HERMES_HOME/SOUL.md`
- 不是一些本地仓库 `SOUL.md`
- 文件不为空
- 编辑后您的会话重新启动
- “/personality”叠加层并未主导结果

### OpenClaw 忽略了我的部分灵魂.md

可能的原因：
- 更高优先级的指令会覆盖它
- 该文件包含相互冲突的指导
- 文件太长并被截断
- 某些文本类似于提示注入内容，可能会被扫描仪阻止或更改

### 我的 SOUL.md 变得过于特定于项目

将项目说明移至“AGENTS.md”，并使“SOUL.md”专注于身份和风格。
